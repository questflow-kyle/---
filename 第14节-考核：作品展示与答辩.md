# 第14节：考核——作品展示与答辩

## 概要

> 课程最后一节，学员展示自己的作品并接受答辩。考核不只看作品本身，更看开发过程中的思维方式——能不能说清楚"做了什么、怎么做的、遇到什么问题、怎么解决的"。考核结果分L1/L2/L3三个等级。

## 教案

### 课时信息

- 所属阶段：考核（第14节/共14节）
- 主讲：全体讲师（沈林伟、梅杭俊、尹伟）
- 时长：45 分钟

### 考核目标

通过作品展示和答辩，评估每位学员达到的等级：
- **L1 能用**：能用AI进行开发工作，能通过搜索和提问解决问题
- **L2 能调**：能理解代码，能有方向地调整
- **L3 能交付**：能完整走通开发→调试→部署全流程

### 考核流程

#### 按5-10人班级规模设计（45分钟）

| 时间 | 环节 | 说明 |
|:---:|:---|:---|
| 0-2分钟 | 开场 | 讲师说明流程和规则 |
| 2-37分钟 | 逐人答辩 | 每人约3-5分钟（展示2-3分钟 + 提问1-2分钟） |
| 37-42分钟 | 讲师合议 | 三位讲师讨论每人等级评定 |
| 42-45分钟 | 结果公布与课程总结 | 公布等级、总结课程、后续学习建议 |

#### 答辩要求

每位学员需完成：
1. **展示作品**：在自己的电脑上运行/打开项目，展示核心功能
2. **讲述过程**：用1-2分钟说清楚：
   - 这个项目做什么、给谁用
   - 开发过程中遇到的一个关键问题是什么
   - 怎么解决的（搜索？问AI？自己改代码？）
3. **回答提问**：讲师根据展示内容提1-2个问题

#### 讲师提问方向

| 考察点 | 示例问题 |
|:---|:---|
| 需求理解 | "你怎么想到做这个的？" |
| 代码理解 | "这段代码是做什么的？如果要改成XX，你会怎么改？" |
| 调试能力 | "开发过程中遇到的最大的坑是什么？你怎么发现问题在哪的？" |
| 工具使用 | "你是怎么跟Claude Code描述这个需求的？" |
| 拓展思考 | "如果要给这个项目加一个新功能，你会加什么？怎么做？" |

### 评分标准

#### L1 · 能用（保底线）

- [ ] 项目能运行，有基本功能
- [ ] 能说出项目做什么
- [ ] 开发过程中用到了AI和搜索来解决问题
- [ ] 了解基本的编程概念（变量、条件、循环能说出大意）

#### L2 · 能调（期望线）

- [ ] 满足L1全部标准
- [ ] 能解释代码的主要逻辑（"这段是读文件的、这段是判断条件的"）
- [ ] 在开发过程中做过有方向的调整（不只是让AI跑通，而是自己决定改哪里）
- [ ] 遇到问题时能缩小范围定位（不是只说"报错了"，而是能说"第X行有问题，可能是因为XX"）

#### L3 · 能交付（优秀线）

- [ ] 满足L2全部标准
- [ ] 项目功能完整，不只是"能跑"而是"能用"
- [ ] 有使用说明或别人能按说明操作
- [ ] 能讲清项目结构（哪些文件、各自做什么）
- [ ] 尝试了部署（本地服务器 / 在线部署 / 打包给别人用）

### 教学提示

- 答辩氛围以鼓励为主，不是挑毛病，而是帮学员看到自己的成长
- 提问时根据学员展示的水平调整难度——L1的学员问基础问题，L2/L3的学员追问深一层
- 如果学员紧张说不出来，可以引导："你最开始是怎么跟Claude Code说的？"
- 等级评定时三位讲师各自打分再合议，避免主观偏差
- 公布结果时强调"这是当前水平的记录，不是终点"，给出后续学习建议

### 后续学习建议（结课时给学员）

- **L1学员**：继续用Claude Code做小项目练手，每周做一个。重点练习"读懂代码"和"修改代码"
- **L2学员**：尝试更复杂的项目（如带数据库的应用、API接口调用），练习独立调试
- **L3学员**：可以尝试接真实的小开发任务，学习Git版本管理、项目协作
