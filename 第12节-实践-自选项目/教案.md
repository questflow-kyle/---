# 第12节：实践 -- 自选项目

## 课时信息

| 项目 | 内容 |
|:---|:---|
| 所属阶段 | 实践课（第4节/共5节） |
| 主讲 | 沈林伟 |
| 辅助 | 梅杭俊、尹伟（全体讲师巡回指导） |
| 时长 | 45 分钟 |
| 前置要求 | 学员已完成第9-11节实践课，能用 Claude Code 独立开发脚本和网页应用 |

## 教学目标

学完本节课，学员能够：

1. 从自身生活或工作中找到一个可以用程序解决的问题
2. 把模糊的需求拆解成具体的功能描述（3句话说清楚：做什么、给谁用、什么功能）
3. 用 Claude Code 开始开发自己的项目，跑出初步可运行的版本
4. **（新）** 向另一个人解释自己的项目，并给别人的项目提出建议（结对互审）

## 核心理念（上课前讲师务必读）

> **本节课有两个关键跨越。**
>
> **跨越一：从"给定任务"到"自己定义问题"。** 前三节实践课，任务都是讲师定的。这节课开始，学员自己决定做什么。这是最接近真实开发的状态——自己发现需求、自己描述、用 AI 实现。讲师角色从"带着做"转为"教练巡场"。
>
> **跨越二：第一次进行结对互审（Peer Review）。** 在35分钟时，学员两两配对，互相展示自己的项目并给出反馈。这不只是"展示"——关键是练习"把自己做的东西解释给别人听"这项能力。能讲清楚自己做了什么，本身就是一种教学能力。
>
> 对于没有想法的学员不要有压力。准备好项目方向清单，让他们有得选。需求描述是本节课最关键的环节——描述清楚了，Claude Code 才能写对。
>
> 项目不要求本节完成。第13节还有时间调试和完善。本节重点是"启动并跑出初版"。
>
> **提醒学员：这个项目就是第14节答辩的作品。**

---

## 分层要求

| 等级 | 完成标准 |
|:---:|:---|
| L1 能用 | 能用中文清晰描述需求，用 Claude Code 跑出初步可运行的版本 |
| L2 能调 | L1 + 能在初版基础上修改功能或修复问题；能解释代码的主要逻辑 |
| L3 能交付 | L2 + 项目功能完整、有使用说明、能交给别人使用 |

## 教学流程

### 第一部分：学员回顾 + 任务说明（5分钟）[0:00-0:05]

#### 学员回顾（2分钟）

**讲师台词：**

> "前三节实践课我们做了三样东西：自动整理文件的脚本、处理考勤数据的工具、还有自己的网页应用。谁来说说，经过这三节课，你觉得用 Claude Code 做东西的流程是什么？"

- 请1位学员简要回顾
- 如果学员说不出来，讲师引导

**预期回答要点：**
- 先想清楚要做什么
- 用中文描述需求
- 让 Claude Code 生成代码
- 运行、测试、不满意就修改

**讲师台词（30秒，过渡到任务说明）：**

> "说得很好。流程你们已经很熟练了。但前面三节课有一个共同点——做什么都是我定的。今天不一样了。"

#### 任务说明（3分钟）

**讲师台词：**

> "今天的任务只有一句话：**解决你自己的一个真实问题。**
>
> 你生活中有没有什么事情是重复做的、很麻烦的？工作里有没有什么东西是每次都要手算或者手动整理的？
>
> 比如：每个月记账很麻烦——做一个记账工具。每次算工资要翻来翻去——做一个工资计算器。想背单词但没好用的工具——做一个单词卡片。
>
> 做什么你自己决定。可以是脚本，也可以是网页。如果你一时没想法也没关系，我准备了一份方向清单供你参考。
>
> **有一件重要的事**：今天做的这个项目，就是第14节答辩的作品。所以选一个你真正想做的东西。
>
> 还有一件新的事——今天最后10分钟，我们要做**结对互审**。你和另一个人互相展示项目，互相提建议。这个我一会儿再说，先选方向。"

---

### 第二部分：选方向 + 写需求（7分钟）[0:05-0:12]

> **设计意图**：这是本节课最关键的环节之一。需求描述的质量直接决定后续开发的效率。给学员足够的时间把需求想清楚、写清楚。

**讲师台词：**

> "现在每个人做三件事：
>
> **第一，选一个方向。** 如果你有自己的想法，太好了，直接用。如果没有，看这个参考清单，选一个。"

**投影展示项目方向参考清单：**

```
项目方向参考 —— 选一个最贴近你生活的

┌──────────┬──────────────────────────────────────┐
│ 生活工具  │ 个人记账本、日程提醒器、生活费分摊计算器  │
├──────────┼──────────────────────────────────────┤
│ 工作工具  │ 考勤统计器、工资核算工具、排班表生成器    │
├──────────┼──────────────────────────────────────┤
│ 学习工具  │ 单词记忆卡片、打字练习器、知识问答小游戏  │
├──────────┼──────────────────────────────────────┤
│ 信息工具  │ 天气查询页面、汇率换算器、倒计时网页     │
└──────────┴──────────────────────────────────────┘

也可以继续完善上节课做的网页应用！
```

**讲师台词（继续）：**

> "**第二，写需求。** 不用写很多，3句话就够：
> 1. 这个东西做什么？
> 2. 给谁用？
> 3. 要实现哪些功能？
>
> 我准备了一个需求描述模板，你可以照着填。"

**投影展示需求描述模板（简化版）：**

```
需求描述 —— 3句话说清楚

1. 做什么：我要做一个 _____________
2. 给谁用：给 _____________ 用的
3. 功能：  要能 _____________，还要能 _____________

例子：
1. 做什么：我要做一个工资计算器
2. 给谁用：给自己算每月工资用的
3. 功能：  要能输入基本工资和加班小时数，还要能自动算出应发工资和扣税后到手工资
```

> "如果你想写更详细，桌上有一张完整的需求描述模板，你可以填那个。但3句话版本就够了。"
>
> "给你们5分钟时间选方向、写需求。写好了就可以开始做了。"

**讲师和助教分工（这5分钟内）：**

| 角色 | 任务 |
|:---|:---|
| 沈林伟 | 在教室前方待命，回答关于方向选择的问题 |
| 梅杭俊 | 走到没有想法的学员旁边，帮他们选方向、梳理需求 |
| 尹伟 | 帮学员把口头描述变成书面需求（"你刚才说的意思是不是这样？"） |

#### 30秒轮流说（2分钟）

**讲师台词（5分钟后）：**

> "好，时间到。现在快速轮一圈——每个人用一句话说说你要做什么。从这边开始。"

**操作：**
- 每人30秒，只说项目名称和一句话描述
- 讲师对每个人给出简短方向建议（一句话即可）
- 示例互动：

> 学员："我想做一个记账的工具。"
> 讲师："好的，做网页版还是脚本版？"
> 学员："网页的吧。"
> 讲师："可以，上节课的经验直接能用。开始吧。"

> 学员："我想做一个排班表。"
> 讲师："好方向。先做最简单的版本——输入人名和日期，输出排班表。别一开始就想做太复杂的。"

**如果有学员还没想好：**

> "没关系，你先做这个 [从清单上指一个]，做着做着可能就有自己的想法了。"

---

### 第三部分：独立开发（23分钟）[0:12-0:35]

> **设计意图**：本节课最长的环节。全体讲师巡场，学员独立开发。重点是让每个人都跑出一个初步可运行的版本。

**讲师台词（开发前提醒）：**

> "开始开发。几个提醒：
> 1. 创建工作目录：`mkdir -p ~/project && cd ~/project`
> 2. 把你写的需求描述直接发给 Claude Code 就行
> 3. 先做最核心的功能，别一开始就想做完美
> 4. 目标是今天结束时有一个能运行的初版
> 5. 一会儿你要把这个东西展示给另一个人看，所以确保它能跑起来
>
> 好，开始！"

#### 全体讲师巡场分工

| 讲师 | 巡场重点 | 站位 |
|:---|:---|:---|
| 沈林伟（主讲） | 技术指导：帮学员解决 Claude Code 使用问题、代码报错、功能实现方向 | 教室中间，哪里需要去哪里 |
| 梅杭俊 | 需求梳理：帮学员把模糊的想法变成清晰的功能描述。重点关注还没开始写代码的学员 | 从后排开始巡 |
| 尹伟 | 环境保障：帮学员解决电脑问题、文件打开问题、Claude Code 不响应等。确保每台电脑都在正常工作 | 哪台电脑有问题就去哪里 |

#### 巡场指导要点

| 观察到的情况 | 指导方式 |
|:---|:---|
| 学员5分钟了还没开始写代码 | "你写的需求给我看看？... 好，这个直接发给 Claude Code 就行，试试看" |
| 需求太大太复杂（比如"做一个完整的ERP系统"） | "这个想法很好，但今天时间只有20分钟。我们先做最核心的一个功能——你觉得哪个最重要？先做那个" |
| 需求太模糊（"做一个好用的工具"） | "好用的工具做什么的？帮你做什么事情？... 好，那就是一个 [具体功能] 工具。你跟 Claude Code 这样说..." |
| Claude Code 生成的代码有 bug | "先看看是什么错误。把报错信息发给 Claude Code，跟它说'运行报错了，帮我修复'" |
| 学员只跑通了 L1 就停了 | "基本功能有了，不错。你觉得还缺什么？比如如果别人要用这个工具，他能看懂怎么用吗？" |
| 学员做得很快 | "厉害！你可以加一些功能——比如数据保存、输入校验、使用说明。让它更完善" |
| 学员想从零换一个方向 | "可以换，但时间不多了。如果新方向比较简单就换，如果比较复杂建议先把当前的做完" |

#### 进度检查点

在开发过程中（大约25分钟时，即 0:30 左右），讲师快速巡一圈，心里记录每个人的进度，为接下来的结对互审做准备：

| 进度 | 预期 | 互审时处理 |
|:---|:---|:---|
| 有初版在运行 | 大部分学员 | 正常参与互审 |
| 还在开发中但有部分功能 | 部分学员 | 可以互审已有的部分 |
| 卡住了，还没跑通 | 个别学员 | 讲师快速帮忙在互审前跑通一个最小版本 |

### 第四部分：结对互审（7分钟）[0:35-0:42]

> **设计意图**：这是本节课的新元素，也是"同伴教学"的核心实践。学员两两配对，互相展示项目并给反馈。关键目标不是评判质量，而是练习"把自己做的东西解释给别人听"——这本身就是一种教学能力。

#### 互审前说明（1分钟）

**讲师台词：**

> "好，停下手里的工作。不管做到什么程度都没关系——能运行就行，哪怕只有最基本的功能。
>
> 现在我们要做一件新的事情，叫**结对互审**。
>
> 什么意思呢？你和旁边的人组成一对。每个人做三件事：
>
> **第一，展示。** 把你的项目展示给对方看——打开网页或者运行脚本，让对方看看你做了什么。
>
> **第二，提问。** 看完对方的项目后，问对方一个问题。什么问题都行。比如'这个数据保存在哪里？'、'你是怎么跟 Claude Code 描述这个需求的？'、'这个功能是怎么实现的？'
>
> **第三，建议。** 给对方一个建议。比如'如果能加一个删除功能就更好了'、'字体可以再大一点'、'如果能显示日期就更方便了'。
>
> 每人3分钟。我来计时。"

**投影展示互审流程：**

```
结对互审 —— 每人3分钟

步骤1：展示（1分钟）
   → 打开你的项目，给对方看。
   → 说清楚："我做的是 ___，它能 ___"

步骤2：提问（1分钟）
   → 对方问你一个问题，你回答。

步骤3：建议（1分钟）
   → 对方给你一个改进建议。

然后换人，对方展示。
```

#### 分组方式

**讲师操作：**

> "就和你旁边的人一组。如果人数是单数，有一组三个人，每人2分钟。"

- 就近配对，不要让学员换座位（浪费时间）
- 如果有学员的项目还没跑通，跟他说："没关系，你给对方看你写的需求描述和已经有的代码，说说你想做什么、卡在哪里"

#### 互审进行中（6分钟）

**讲师计时：**

> "第一个人开始展示。3分钟之后换人。"

**计时提醒：**
- 1分30秒时："提醒一下，请开始提问和建议环节。"
- 3分钟时："好，换人。第二个人开始展示。"
- 4分30秒时："提醒一下，请开始提问和建议环节。"
- 6分钟时："好，互审结束。"

**讲师和助教在互审期间的角色：**

| 角色 | 任务 |
|:---|:---|
| 沈林伟 | 在各组之间走动，听学员怎么解释自己的项目。如果学员说不清楚，轻声提示："你先说它是做什么的，再说怎么用" |
| 梅杭俊 | 关注有没有一组陷入沉默的——如果两个人都不知道怎么提问，帮忙引导："你觉得他这个工具方便吗？有没有什么功能你觉得可以加上去的？" |
| 尹伟 | 帮助技术问题——如果互审时发现某个项目跑不了，快速帮忙修复 |

#### 互审后简短分享（可选，如果时间充裕，30秒）

> "有没有谁收到了一个特别好的建议，想跟大家分享一下？"

---

### 第五部分：状态检查（3分钟）[0:42-0:45]

> **设计意图**：快速了解每个人的进度，为第13节做准备。

**讲师台词：**

> "最后一分钟。每个人用一句话说说你现在的状态。三种选择：
> 1. **跑通了** —— 我的项目能运行了
> 2. **卡在了** —— 我卡在 XXX 上
> 3. **下一步** —— 我的下一步是做 XXX
>
> 从这边开始，每人一句话。"

**操作：**
- 每人一句话，快速轮过去
- 讲师记录谁卡在哪里（下节课开头可以重点帮助）
- 不评论、不展开讨论，纯粹收集状态

**讲师总结（30秒）：**

> "今天你们做了一件很不一样的事——不是我告诉你做什么，而是你自己定义问题、自己解决。这就是真正的开发。
>
> 而且你们刚才互相展示了自己的项目，给了别人建议。把自己做的东西讲清楚、听取别人的意见——这个能力和写代码一样重要。
>
> 下节课是最后一节实践课——调试、完善、交付。你今天的项目还可以继续改。带着互审时收到的建议去改进它。"

## 结对互审 -- 讲师指导手册

> 这是本节课新引入的环节，以下是详细的操作指导。

### 为什么要做结对互审？

1. **练习解释能力**：能把自己做的东西讲清楚，是一种重要的职业技能
2. **练习倾听和反馈**：学会从别人的视角看问题
3. **为第14节答辩做准备**：答辩时要向所有人展示并回答问题，结对互审是小规模预演
4. **发现改进空间**：自己看不到的问题，别人一眼就能看到

### 可能遇到的问题和应对

| 问题 | 应对 |
|:---|:---|
| 学员不知道怎么提问 | 讲师给示范问题："你可以问——这个工具解决了什么问题？你是怎么跟 AI 描述需求的？如果我是用户，我第一步该做什么？" |
| 学员不知道怎么给建议 | 讲师引导："你觉得这个工具好用吗？如果你来用，有什么地方不方便？有什么功能你觉得可以加上去？" |
| 学员对自己的项目不好意思展示（觉得太简单） | "没关系，简单不等于不好。一个能用的简单工具，比一个不能用的复杂工具强一百倍。你就给对方看看，说说你想做什么。" |
| 互审变成了闲聊 | 讲师走过去轻声提醒："时间不多，先完成展示和提问再聊。" |
| 学员的项目还没能运行 | "你给对方看你写的需求描述。说说你想做什么、目前做到哪里了、卡在哪里。这本身也是一种展示。" |

### 好的互审示范对话

讲师在说明互审规则时，可以快速演示一段：

> **讲师模拟展示方：** "我做了一个工资计算器。你输入基本工资和加班小时数，它会算出你的应发工资。来，你试试。"
>
> **讲师模拟提问方：** "如果我加班超过36小时怎么办？法律规定每月加班不能超过36小时，你这个会提醒吗？"
>
> **讲师模拟展示方：** "现在还没这个功能。"
>
> **讲师模拟提问方：** "那我建议加上去。如果输入超过36就给一个提示。这样工具更实用。"
>
> **讲师总结：** "看到了吗？就是这样。展示、提问、建议。很简单，不需要很正式。"

## 常见问题与应对（FAQ）

| 学员问题 | 讲师回应 |
|:---|:---|
| "我不知道做什么项目" | "看看参考清单。或者想想你每天最烦的一件事——有没有办法用程序解决？实在没想法的话你就做这个 [从清单上指一个]" |
| "我上节课做的网页能继续用吗？" | "当然可以！在上节课的基础上加功能、做完善，这就是你的项目。" |
| "我的项目太简单了怎么办？" | "没有'太简单'这回事。一个能用的简单工具就是好工具。先做好基本功能，有时间再加功能。" |
| "我的项目太复杂了，做不完怎么办？" | "不需要今天做完。今天的目标是跑出一个初版。下节课还有时间完善。先做最核心的一个功能。" |
| "我想做一个手机App" | "好想法，但手机App需要更多工具和时间。建议先做成网页——手机浏览器也能打开网页，效果差不多。" |
| "两个人都做一样的方向可以吗？" | "可以。同样的方向，每个人的需求不一样，做出来肯定不一样。" |
| "需求描述要写几句话？" | "至少3句：做什么、给谁用、什么功能。能写更详细更好，但3句话是最低要求。" |
| "结对互审要打分吗？" | "不打分。目的是练习解释你做了什么、听取别人的意见。没有对错好坏。" |

## 时间应急方案

| 情况 | 方案 |
|:---|:---|
| 选方向 + 写需求超时（超过12分钟） | 砍掉30秒轮流说环节，直接让学员开始开发。讲师在巡场时一对一确认每人的方向 |
| 大部分学员在20分钟时还没跑出初版 | 压缩互审时间到4分钟（每人2分钟），让学员多一些开发时间 |
| 互审环节冷场 | 讲师演示一遍互审过程（30秒），然后走到每一组旁边引导："来，你先给他看看你做了什么" |
| 时间多余（开发提前完成） | 延长互审时间，让学员做两轮互审——换一个人再审一遍。或让完成的学员帮卡住的同学 |
| 有学员从头到尾都卡住了 | 讲师或助教在互审环节之前帮他跑出一个最小版本（哪怕只是一个显示文字的网页也行） |

课上可以在白板或投影上保留以下内容供学员随时参考：

```
今天的任务：解决你自己的一个真实问题

需求描述 3 句话：
  1. 做什么：我要做一个 ___________
  2. 给谁用：给 ___________ 用的
  3. 功能：  要能 ___________

操作流程：
  mkdir -p ~/project && cd ~/project
  claude
  → 把需求描述发给 Claude Code
  → 运行、测试、修改

结对互审 3 步：
  1. 展示 → "我做的是___，它能___"
  2. 提问 → 问对方一个问题
  3. 建议 → 给对方一个改进建议

这个项目 = 第14节答辩作品！
```
