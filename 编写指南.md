# 计算机通识课 —— 教案编写指南

> 本文件是所有教案编写agent的统一参考。编写任何一节课的教案或课前检查清单前，请先通读本文档。

## 一、组织背景与战略目标

### 赤耘是谁

赤耘是一个服务无产青年的组织。计算机通识课是通识培训体系（计算机、英语、体育、公文写作）的一部分。

### 战略目标

- 全方位提升人员的专业能力，为开发、外贸、公文等未来工作做储备
- **自我复制的教学系统**：学员学完后能回来担任下一期的老师

### 课程定位

面向会基本电脑操作的学员，统一培养目标：**每个人都能用AI进行开发**。通识内容（搜索、办公、硬件、安全）保留但压缩，为AI开发能力让出课时。

## 二、学员画像与教师画像

### 学员画像

- **身份**：无产青年，白天上班晚上学习，时间极度紧张
- **起点**：会基本电脑操作（开机、浏览器、打字），无编程基础
- **学习动机**：提升职业能力，为未来工作（开发、外贸、公文）做储备
- **特殊要求**：学完后要能回来当老师（所以必须真正理解，不能只是模仿）

### 教师画像

- **身份**：志愿者，白天上班晚上教课，同样时间紧张
- **可能情况**：上一期的学员，刚学完就来教课
- **核心需求**：拿到教案就能直接讲，不需要额外备课时间

## 三、课程基本信息

| 项目 | 内容 |
|:---|:---|
| 授课形式 | 线下面授，隔天一节 |
| 课时标准 | 每节课 45 分钟 |
| 学员管理 | 5-10 人小班 |
| AI工具策略 | 基础阶段用网页版AI → 第5节起引入Claude Code |

### 考核等级

| 等级 | 关键词 | 标准 |
|:---:|:---:|:---|
| L1 | **能用** | 能用AI进行开发工作；能通过搜索、提问解决大部分问题 |
| L2 | **能调** | 能理解AI生成的代码；能有方向地调整代码逻辑和功能 |
| L3 | **能交付** | 能完整走通开发→调试→部署全流程；能独立交付可运行的成品 |

- **L2 是期望线**：所有学员应达到
- L1 是保底，L3 是优秀

## 四、教案格式规范（剧本式结构模板）

### 核心原则

教案的详细程度要达到 **"一个刚学完的学员拿到手就能当老师讲"** 的水平。

### 文件头

```markdown
# 第X节：课程名称 —— 剧本式教案

## 课时信息

| 项目 | 内容 |
|:---|:---|
| 所属阶段 | 基础课/实践课/考核（第X节/共14节） |
| 主讲 | 讲师姓名 |
| 辅助 | 助教姓名（如有） |
| 时长 | 45 分钟 |
| 班级规模 | 5-10 人 |
| 前置要求 | 上一节课的名称 |

## 教学目标

学完本节课，学员能够：
1. ...
2. ...
3. ...
4. ...
```

### 时间分配表

每份教案必须包含一个时间分配总览表：

```markdown
## 时间分配

| 环节 | 时长 | 累计 |
|:---|:---:|:---:|
| 开场/回顾 | X分钟 | 0:XX |
| 环节一 | X分钟 | 0:XX |
| ... | ... | ... |
| 总计 | 45分钟 | 0:45 |
```

### 每个环节的必备要素

每个教学环节必须包含以下四个部分：

#### 1. 讲师说（对话示例）

用引用块写出讲师要说的原话，不是提纲，是**逐字稿**。

```markdown
### 讲师说

> "这段话是讲师要说的原话，用口语化的表达，不要书面语。
> 要说得像平时聊天一样自然。"
```

#### 2. 讲师做（动作指示）

用列表写出讲师在这个环节要做的具体操作。

```markdown
### 动作

- 打开投影，展示XXX页面
- 走到学员中间，确认每个人都看得到屏幕
- 在投影上演示：输入XXX → 点击XXX → 展示结果
```

#### 3. 学员应看到什么（预期画面/结果）

```markdown
### 学员屏幕

学员此时应该看到：
- 浏览器打开了XXX页面
- 搜索结果中前3条是XXX
```

#### 4. 常见卡点与应对

```markdown
### 可能的卡点

| 现象 | 原因 | 讲师应对 |
|:---|:---|:---|
| 学员屏幕上没有出现XXX | 可能是XXX | 走过去检查，帮学员XXX |
```

### 练习环节格式

```markdown
## 练习（X分钟）

### 任务说明

| 序号 | 任务 | 完成标准 |
|:---:|:---|:---|
| 1 | 做XXX | 看到XXX说明成功 |

### 讲师说

> "现在大家来练习。任务是XXX。先完成的同学帮一下旁边还没完成的。"

### 巡场要点

- 重点观察：学员是否XXX
- 先完成的学员：引导去帮助未完成的同学
- 如果大部分人卡在同一个地方：暂停练习，统一讲解
```

### FAQ 部分（每节课必备）

每节课末尾必须包含 3-5 个学员最可能问的问题及回答：

```markdown
## FAQ

### Q1：XXX？

> **讲师回答**："XXX"

### Q2：XXX？

> **讲师回答**："XXX"
```

### 课后小结格式

```markdown
## 课后小结（2分钟）

### 讲师说

> "今天我们学了三件事：第一XXX，第二XXX，第三XXX。
> 下节课我们会XXX。"

### 布置课后任务（可选）

- 任务描述
- 完成标准
```

## 五、同伴教学机制

从第3节起，每节课必须包含以下同伴教学元素：

### 1. 开场学员回顾（5分钟）

第3节起，每节课开头安排一名学员主讲上节课的内容回顾（5分钟）。

```markdown
## 开场：学员回顾（5分钟）

### 机制

- 上节课结束时指定下节课负责回顾的学员（轮流制）
- 学员用自己的话讲上节课学了什么，2-3分钟
- 讲师补充和纠正，1-2分钟

### 讲师说（引导）

> "今天先请XXX来给大家回顾一下上节课的内容。XXX，你来说说上节课我们学了什么？"

### 讲师注意

- 学员讲得不完整没关系，重点是让他们开口
- 如果学员紧张说不出来，用提问引导："上节课我们用了什么工具？""做了什么练习？"
- 讲完后一定要肯定："说得很好"或"方向对了，我补充一点……"
```

### 2. 练习环节的互助

所有练习环节必须写入 **"先完成的帮未完成的"** 机制。

## 七、各节课的具体改良要求

> 以下是每节课在原有大纲基础上需要重点改良的方向。编写教案时必须落实。

| 节次 | 主题 | 改良重点 |
|:---|:---|:---|
| 第1节 | 搜索与信息素养 | **重写**。以"找工作"场景贯穿全课，整合渠道意识+搜索方法+信息判断三问。AI作为验证渠道自然引入（点到为止）。核心能力：信息判断三问（谁在说、真不真、为什么） |
| 第2节 | AI对话与提问技巧 | **改良**。开头衔接第1节（学员已体验过AI），从"学一个新工具"改为"深入学AI"。幻觉环节显式回调三问法，建立"同一套思维工具换场景照样能用"的认知。课后小结指定第3节回顾人选 |
| 第3节 | 办公基础与AI提效 | **改良**。砍掉邮件内容，聚焦Word+Excel+AI协作。练习场景统一为仓库管理员简历（延续L1/L2主线）。AI生成内容环节加三问法轻触回调。本节起引入同伴教学（开场学员回顾） |

| 第9节 | 实践：自动化脚本 | 基本保持结构，补充剧本式细节 |

| 第11节 | 实践：网页应用开发 | 基本保持结构，补充剧本式细节 |
| 第12节 | 实践：自选项目 | 加入结对互审环节 |
| 第13节 | 实践：调试部署与交付 | 基本保持，补充剧本式细节 |
| 第14节 | 考核：作品展示与答辩 | 补充完整的评分量表和提问题库 |

### 第1节特别说明

本节课已重写，核心设计变更：

- **砍掉**原有的"信息来源层级"（官方>新闻>自媒体）这种口号式教法
- **新增渠道意识**：不同需求去不同渠道，每个平台的商业逻辑决定了它展示什么信息
- **新增信息判断三问**（谁在说、能否验证、动机是什么）：通过真实案例（虚假高薪招聘帖）建立结构性追问习惯，这是全课程的底层思维框架
- **AI自然引入**：在案例分析和综合练习中，AI作为信息验证渠道之一出现（点到为止），为第2节做铺垫
- **搜索技巧压缩**：只保留引号和site:两个技巧，砍掉filetype:
- **贯穿场景**：全课围绕"在成都找一份仓库管理员的工作"展开

### 第2节特别说明

本节课已改良，核心设计变更：

- **开头衔接第1节**：不再以"认识AI"开场，而是"上节课已经体验过AI，今天深入学"。开场时回调三问法（谁在说、真不真、为什么），检验学员是否记住
- **AI定位讲解增加原理**：用"预测下一个最可能出现的字"这个大白话解释AI的工作原理，为后面理解幻觉做铺垫
- **幻觉环节新增三问法回调**：演示幻觉后，显式用第1节的三个问题分析AI——谁在说（语言模型）、能否验证（搜索）、动机（不是骗你，是讨好你）。这是有机整合的关键连接点，让三问法从"第1节的知识点"升级为"跨场景的思维习惯"
- **课后小结新增指定回顾人选**：第3节起启用同伴教学（学员主讲回顾），因此第2节结尾必须指定人选，这是第3节的前置条件
- **练习任务延续找工作场景**：任务2的简历优化延续了第1节"找仓库管理员工作"的贯穿场景，形成链路：第1节搜索职位 → 第2节AI写简历 → 第3节Word排版简历
- **格式补齐**：各环节补充了学员屏幕、可能的卡点等标准要素；课后记录模板统一为课后三问格式

### 第3节特别说明

原大纲包含邮件内容（写邮件、邮件礼仪、文件分享），现在**全部砍掉**。45分钟全部给 Word + Excel + AI协作：

- Word部分（~15分钟）：新建保存、文字排版、段落格式、导出PDF、AI生成内容→自己排版
- Excel部分（~15分钟）：单元格操作、SUM/AVERAGE公式、排序筛选、AI写公式→人验证
- 练习（~8分钟）：用AI生成简历内容→检查真伪→Word排版→导出PDF；Excel录入数据用公式计算
- 开场回顾（5分钟）：本节起引入学员主讲回顾
- **场景延续**：练习场景统一为"仓库管理员"求职简历，延续第1节（搜职位）→第2节（AI写简历）→第3节（排版简历）的完整链路
- **三问法轻触回调**：AI生成简历内容时，自然提醒"AI会编造经历，你自己最清楚是真是假"；AI写公式时，手算验证体现"信但验"。不重新讲三问法，点到即止
- **人机协作模式建立**：本节课建立"AI生成→人验证/排版"的协作模式，这个模式在后续课程中会深化为"AI写代码→人读懂/修改"

### 第4节特别说明

本节课已改良，核心设计变更：

- **修正学员回顾要点**：删除了对已砍掉的邮件内容的引用，更新为L3实际教授的内容（Word排版+Excel+AI协作+信但验）
- **Claude Code与信但验的连接**：引入Claude Code时，显式点明"它本质上是同一个AI，只是多了操作电脑的能力。能力更强，验证就更重要"。这为第5节以后"AI写代码→人验证"的协作模式做铺垫
- **设计说明**：明确本节课是"信息素养+AI+办公"到"编程+开发"的桥梁，前三节的能力（三问法、提问三要素、信但验、人机协作模式）从本节课开始向开发领域迁移
- **课后小结改为必做**：从"如果时间允许"改为正式的2分钟环节，包含阶段总结和指定下节课回顾人选
- **课后记录模板统一**：改为课后三问格式，与L1-L3保持一致

### 第5节特别说明

本节课已改良，核心设计变更：

- **格式标准化**：去除YAML frontmatter，统一为剧本式教案标准格式（课时信息表、设计说明、标准节标签）。修正"共8节"为"共14节"，补充班级规模
- **设计说明新增**：阐述本节课是人机协作模式的第一次升级——第3节"AI写内容→人排版验证"升级为"AI写代码→人运行读懂修改"。角色分工不变，复杂度提升
- **信但验在编程中的体现**：读懂AI写的代码 = 验证AI的输出。在"一起读代码"环节开头显式连接第3节经验："第3节你们检查AI写的简历有没有编造经历，现在你们检查AI写的代码是不是在做你想要的事情"
- **环节二连接前面课程**："还记得前几节课的方法吗？AI帮你写简历内容，你来排版验证；AI帮你写Excel公式，你来手算验证。今天开始，AI帮你写代码，你来运行、读懂、修改。"让学员意识到方法是连贯的
- **学员回顾要点更新**：补充L4新增的信但验连接（Claude Code本质是同一个AI，能力更强验证更重要）
- **课后小结独立为正式2分钟环节**：包含方法回顾、三个新词总结、信但验提醒、指定下节课回顾人选
- **课后记录模板统一**：改为课后三问格式
- **练习时间微调**：从15分钟调整为13分钟，腾出2分钟给课后小结

### 第8节特别说明

本节是基础课（第1-8节）的收官，需要加入一个**中间检查点**：

- 用一个小任务（5-8分钟）检验学员对前7节内容的掌握
- 检查点任务示例：给一段文字需求，学员独立完成"搜索信息→用AI处理→在终端运行代码查看结果"的完整链路
- 根据检查结果调整后续实践课的节奏

## 八、跨课程设计原则

> 以下原则贯穿整个课程体系，编写任何一节教案时都需要有意识地落实。

### 1. 有机整合，不是模块拼接

课程的14节课不是14个孤立的知识模块，而是一个有机整合的能力培养体系。每节课的内容应当自然地调用前面课程建立的能力，同时为后续课程铺路。

**核心原则：1 + 1 + 1 >>> 3**

- 不同课程的知识点应当在实际场景中交织出现，而不是在各自的课时内封闭循环
- 例：第1节的"信息判断三问"不只在第1节使用，它在第2节（判断AI是否幻觉）、第3节（检查AI生成的简历内容）、第8节（阶段检查）、实践课（验证代码运行结果）中反复调用
- 例：AI工具不是在第2节才"突然出现"，而是在第1节就作为信息验证渠道之一自然出场，第2节深入学习其用法

### 2. 信息判断三问：贯穿全课程的底层思维框架

第1节建立的"信息判断三问"（谁在说、能否验证、动机是什么）是整个课程体系的底层思维框架。后续课程在涉及信息判断、结果验证时，应当有意识地回调这个框架：

| 课程 | 三问的具体应用 |
|:---|:---|
| 第1节 | 判断招聘信息、社保消息的可信度 |
| 第2节 | 判断AI回答是否可信——AI是"谁在说"？它说的能验证吗？ |
| 第3节 | 检查AI生成的简历内容——AI写的经历是真的吗？ |
| 第8节 | 阶段检查中验证搜索结果和代码运行结果 |
| 第9-13节 | 验证AI生成的代码是否正确——运行结果对不对？逻辑说得通吗？ |

讲师不需要每节课都重新讲三问法，但在涉及验证环节时应当自然地提及："还记得我们说的——先验证再信。"

### 3. AI渐进引入策略

AI工具在课程中采取渐进引入的策略，而不是在某一节课"从零开始"：

```
第1节：AI作为信息渠道之一自然出场（点到为止，只用来验证一个数据）
第2节：系统学习AI对话技巧（提问框架、多轮对话、幻觉识别）
第3节：AI生成内容 + 人类排版/验证（办公场景）
第4节：Claude Code初体验（从网页版升级到终端版，信但验原则不变）
第5-7节：AI生成代码 + 人类读懂/修改（编程场景，人机协作模式升级）
第8节：基础课收官 + 中间检查点
第9-13节：Claude Code作为核心开发工具
```

编写前面的课程时，不要把AI当作"还没教过的东西"来回避；编写后面的课程时，不要把AI当作"刚刚学的新工具"来重新介绍。

### 4. 渠道意识与平台商业逻辑

第1节建立的渠道意识（不同需求去不同平台、每个平台有其商业逻辑）是学员的信息素养基础。后续课程在涉及信息获取时，应当延续这个意识：

- 搜索引擎、AI、官方文档、社区论坛——都是不同的信息渠道
- 每个渠道有优势也有局限性，没有万能渠道
- 交叉验证的习惯应当贯穿始终

## 九、通用改良要求（所有节次）

编写每节课教案时，以下要求必须全部落实：

1. **剧本式格式**：每个环节写明讲师要做什么、说什么（对话示例/逐字稿）、学员应看到什么
2. **同伴教学**（第3节起）：每节课开头5分钟回顾由学员主讲；练习环节写入"先完成的帮未完成的"
3. **FAQ**：每节课 3-5 个学员最可能问的问题及回答
4. **课前检查清单**：独立文件（`课前检查与材料.md`），课前15-30分钟逐项检查
5. **可复制性**：教案详细程度 = "一个刚学完的学员拿到手就能当老师讲"
6. **时间分配表**：每节课有完整的时间分配总览
7. **备用方案**：关键环节有故障应对方案（写在课前检查清单中）

## 十、写作风格要求

1. **口语化**：讲师说的话要像平时聊天，不要书面语
2. **具体化**：不写"演示搜索"，要写"在百度搜索框输入'成都 2026 最低工资标准'，按回车"
3. **场景化**：练习用学员生活中真实会遇到的场景（请假条、简历、工资计算等）
4. **不假设知识**：每个术语第一次出现时用大白话解释
5. **投影内容写出来**：需要投影展示的内容（表格、代码、提示词等）直接写在教案/课前检查文件中，不要只写"准备PPT"
