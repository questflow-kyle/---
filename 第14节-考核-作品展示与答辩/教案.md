# 第14节：考核 -- 作品展示与答辩

## 课时信息

| 项目 | 内容 |
|:---|:---|
| 所属阶段 | 考核（第14节/共14节） |
| 主讲 | 全体讲师（沈林伟、梅杭俊、尹伟） |
| 时长 | 45 分钟 |
| 班级规模 | 5-10人 |
| 前置要求 | 学员已完成第13节，项目已打磨至可展示状态 |

## 考核目标

通过作品展示和答辩，评估每位学员在14节课中达到的能力等级：

- **L1 能用**：能用AI进行开发工作，能通过搜索和提问解决问题
- **L2 能调**：能理解代码，能有方向地调整和调试
- **L3 能交付**：能完整走通开发 → 调试 → 部署全流程

## 核心理念（全体讲师务必读）

> **答辩的目的是鼓励，不是淘汰。**
>
> 这些学员从零开始，14节课做出了自己的项目，这本身就值得肯定。答辩环节要让每个人都感受到"我学到了东西、我做出了成果"，而不是"我还差很远"。
>
> **提问时根据学员展示的水平调整难度**——L1的学员问基础问题，L2/L3的学员追问深一层。不要用L3的标准去要求L1的学员。
>
> **等级评定是记录当前水平，不是盖棺定论。** 公布结果时要强调"这是起点，不是终点"。
>
> 讲师分工：
> - 沈林伟：主持答辩流程，控制时间
> - 梅杭俊：记录评分，观察学员表现
> - 尹伟：技术层面的提问，协助学员电脑操作

---

## 考核流程

### 开场（2分钟）[0:00-0:02]

**沈林伟台词：**

> "好，今天是我们这门课的最后一节。14节课，从搜索开始，到AI对话、到办公软件、到开发环境、到写程序、到做项目——大家走了很长一段路。"
>
> "今天的答辩不是考试。你们每个人都做出了自己的作品，今天就是把它展示出来，跟我们聊聊你做了什么、怎么做的。"
>
> "流程很简单："

**投影显示：**

```
答辩流程

每人 3-5 分钟：
1. 展示你的项目运行（2-3分钟）
2. 回答讲师1-2个问题（1-2分钟）

你需要说清楚三件事：
 ① 这个项目做什么、给谁用
 ② 开发过程中遇到的一个关键问题
 ③ 你怎么解决的

放轻松，就当跟我们聊天。
```

**沈林伟台词：**

> "谁想第一个来？主动来的加印象分。"

**如果没人主动：**

> "那我来点名了。（选一个平时表现比较积极、不容易紧张的学员）XXX你先来，给大家打个头阵。"

---

### 逐人答辩（35分钟）[0:02-0:37]

> **时间控制**：5人班级每人约5分钟（展示3分钟+提问2分钟），10人班级每人约3.5分钟（展示2分钟+提问1.5分钟）。沈林伟负责控时，快到时间用手势提醒。

#### 每位学员的答辩流程

**第一步：展示作品（2-3分钟）**

学员在自己的电脑上运行/打开项目，向大家展示核心功能。

**沈林伟引导（如果学员不知道从哪开始）：**

> "你先把程序跑起来给我们看。然后跟我们说说：这个程序是做什么的？你为什么想做这个？"

**如果学员紧张说不出来：**

> "没关系，你先运行程序。它运行起来了对吧？好，那你告诉我，你输入了什么，它输出了什么？这个功能你觉得有用吗？"

**如果程序无法运行：**

> "没关系，程序有时候就是这样。你打开代码给我们看看，跟我们说说你写了什么功能。就算没跑通，你的思路和过程也是有价值的。"

**第二步：讲师提问（1-2分钟）**

三位讲师根据学员展示的水平，从答辩提问库中选择合适的问题。

**提问原则：**

| 学员表现 | 提问策略 |
|:---|:---|
| 项目简单、展示紧张 | 问最基础的问题，帮学员说出来。比如"你是怎么跟Claude Code说的？" |
| 项目能跑、能基本解释 | 追问代码理解和调试过程。比如"这段代码是做什么的？遇到的最大bug是什么？" |
| 项目完整、表达清楚 | 问拓展性问题。比如"如果要加新功能你会怎么做？为什么这样设计？" |

**讲师提问分工：**

- 沈林伟：主要问需求理解和过程方面的问题
- 梅杭俊：主要观察和记录，偶尔补充提问
- 尹伟：主要问技术层面的问题（代码理解、工具使用）

**第三步：过渡到下一位**

**沈林伟台词（每位学员结束时）：**

> "好，谢谢XXX！（可以简短点评一句亮点）下一位，XXX你来。"

**简短点评示例：**

| 情况 | 点评 |
|:---|:---|
| 项目简单但跑通了 | "能跑通就是很好的开始，做出来了就是胜利。" |
| 遇到问题但解决了 | "解决问题的过程比项目本身更重要。你这个调试思路很好。" |
| 项目完整度高 | "项目做得很完整，看得出花了心思。" |
| 表达清楚 | "你解释得很清楚，能把代码讲明白这件事本身就很厉害。" |

#### 答辩注意事项（全体讲师）

1. **不要打断学员**：学员展示的时候让他说完，即使跑题了也先听完再引导
2. **不要纠正细节**：学员说的概念不完全准确没关系，答辩不是纠错课
3. **不要比较学员**：不要说"你看XXX做得比你好"这类话
4. **鼓励为主**：每个学员结束时至少说一句肯定的话
5. **紧张的学员多引导**：如果学员卡壳了，用问题引导他说出来，而不是跳过
6. **控制时间**：每人不超过5分钟。快到时间时给手势提醒，提问环节最多问2个问题

---

### 讲师合议（5分钟）[0:37-0:42]

> 学员可以休息、互相交流。三位讲师到一边讨论评级。

**沈林伟台词：**

> "大家展示完了，辛苦了！现在休息5分钟，我们三位老师商量一下。"

#### 合议流程

1. **各自打分**：每位讲师先独立看自己的评分表，确认每位学员的等级
2. **对齐标准**：三人核对结果，如有分歧，讨论具体证据（"他能解释代码逻辑吗？""她调试的时候是自己读报错还是让AI修？"）
3. **最终定级**：以多数意见为准。边界情况（如L1和L2之间）就高不就低——鼓励为主

**评级参照标准：**

| 等级 | 核心判断依据 |
|:---|:---|
| L1 能用 | 项目能运行（哪怕很简单），能说出做什么，开发中用了AI和搜索 |
| L2 能调 | 在L1基础上，能解释代码主要逻辑，做过有方向的调整（不只是"AI帮我做的"） |
| L3 能交付 | 在L2基础上，功能完整、有使用说明、能讲清项目结构、尝试了部署 |

**特殊情况处理：**

| 情况 | 处理方式 |
|:---|:---|
| 程序无法运行但过程清楚 | 可以给L1。过程理解也有价值 |
| 程序是纯复制的，学员无法解释 | 需要确认。如果确实无法解释任何部分，酌情降级 |
| 学员发挥失常（平时好但答辩紧张） | 结合平时课堂表现综合判定，不完全依赖答辩表现 |

---

### 结果公布与课程总结（3分钟）[0:42-0:45]

#### 公布结果

**沈林伟台词：**

> "好，结果出来了。再说一遍：这个等级是记录你**现在**的水平，不是给你贴标签。每个人的起点不同，进步的速度也不同。重要的是你比14节课之前的自己强了。"
>
> "我来公布结果。"

**公布方式：**

逐个公布，每个人都说一句肯定的话：

> "XXX——L2。你的调试能力进步很大，从不知道怎么看报错到现在能自己定位问题，这个变化很明显。"
>
> "XXX——L1。你从完全不会编程到现在能用AI做出一个工具，这已经是很大的突破了。"
>
> "XXX——L3。你的项目完成度很高，还写了使用说明。这个水平可以做真正的小开发了。"

**关键原则：每个人都要被肯定。不要只表扬L3，L1同样值得鼓励。**

#### 课程总结

**沈林伟台词：**

> "14节课结束了。我们来回顾一下你们学了什么。"
>
> "第一阶段，你们学了搜索、AI对话、办公软件——这些是基本的信息素养。"
>
> "第二阶段，你们接触了终端、Python、Claude Code——这些是开发工具。"
>
> "第三阶段，你们自己做了项目——自动化脚本、数据处理、网页应用、自选项目。"
>
> "14节课前，你们可能觉得编程是另一个世界的事情。现在你们知道了：说清楚你要什么，AI帮你写代码，你来读、来改、来调试。这就是这个时代的编程方式。"

#### 后续学习建议

**沈林伟台词：**

> "课程结束了，但学习不用停。根据你们不同的水平，我给几个建议。"

**投影显示：**

```
后续学习建议

L1 的同学：
  - 每周用 Claude Code 做一个小项目，什么都行
  - 重点练习：读懂代码、修改代码
  - 遇到问题先搜索、先读报错，再问AI

L2 的同学：
  - 尝试更复杂的项目（带数据库的、调API的）
  - 练习独立调试：不依赖AI，自己先读报错定位问题
  - 尝试看别人的代码，学习不同的写法

L3 的同学：
  - 可以尝试接真实的小开发任务
  - 学习 Git 版本管理和团队协作
  - 把你的项目放到 GitHub 上，开始建立自己的作品集

所有人：
  - 保持好奇心，不断尝试新东西
  - 编程是手艺活，做得越多越熟练
  - 考虑回来当下一期的助教——教别人是最好的学习方式
```

**沈林伟台词（最后的话）：**

> "最后一件事。这门课结束了，但如果你们以后做项目遇到问题，随时可以来找我们。"
>
> "还有一个想法：下一期课程的时候，你们中间愿意的人可以回来当助教。教别人是最好的学习方式，你们走过的路，可以帮下一期的同学走得更顺。"
>
> "好，恭喜大家完成了赤耘计算机通识课。谢谢！"

## 常见问题与应对（FAQ）

| 情况 | 讲师回应 |
|:---|:---|
| 学员程序答辩时突然跑不了 | "别着急，重新运行一次试试。如果还是不行，你打开代码给我们看，跟我们讲讲思路。程序暂时有问题不影响你的评分，关键看你的理解。" |
| 学员非常紧张，说不出话 | "深呼吸。你就当跟我们聊天。你先运行一下程序给我们看，然后我来问你几个问题，你就回答就行。不用想着'正确答案'。" |
| 学员对评级结果不满 | "这个等级反映的是现在的水平。你继续练习，水平一定会提高。L1到L2的关键是'能读懂代码、能自己定位问题'，你可以朝这个方向努力。" |
| 学员互相比较 | "每个人起点不同，不要横向比较。你跟14节课之前的自己比，进步了多少？这才是最重要的。" |
| 某位学员的项目明显是照抄的 | 提问时重点问"这段代码做什么""如果要改成XXX怎么改"。如果完全无法回答，合议时讨论是否是自己做的。但不要当场质疑学员，保护其自尊心。 |
| 时间不够，后面几位学员来不及 | 缩短提问环节到1个问题，展示环节控制在2分钟。合议环节也可以缩短到3分钟。 |

## 时间应急方案

| 情况 | 方案 |
|:---|:---|
| 只有5人：时间充裕 | 每人可以5分钟，提问可以多1-2个。课程总结环节可以展开讲 |
| 有8-10人：时间紧 | 每人严格3.5分钟（展示2分钟+提问1.5分钟），提问只问1个。合议缩短到3分钟 |
| 有学员缺席 | 时间分配不变，把多出来的时间给课程总结和后续建议 |
| 投影仪故障 | 让学员在自己电脑上展示，其他人围过来看（适合小班）。或让学员口述功能 |
| 某学员展示时间过长 | 沈林伟做手势提醒。如果还在继续，口头说"时间差不多了，最后说一句" |
